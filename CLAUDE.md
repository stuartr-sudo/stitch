# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run start      # Express API (port 3003) + Vite dev server (port 4390) concurrently
npm run server     # Express API only (port 3003)
npm run dev        # Vite frontend only (port 4390)
npm run build      # Production build → dist/
npm run preview    # Preview production build locally (Vite preview)
fly deploy         # Deploy to Fly.io (Sydney region)
```

No test runner or linter is configured.

## Architecture

Full-stack AI video ad creator. React 18 frontend talks to an Express API backend, both backed by Supabase (Postgres + Auth + Storage). Video/image generation delegates to Wavespeed, FAL.ai, and OpenAI. TTS voiceover uses Gemini TTS via FAL.ai proxy (default) or legacy ElevenLabs via FAL.ai proxy.

**Frontend** (`src/`): Vite + React + Tailwind + Radix UI. Path alias `@` → `src/`. All API calls go through `src/lib/api.js` which injects Supabase JWT automatically. Auth state managed via `src/contexts/AuthContext.jsx`. UI components follow shadcn/ui patterns (CVA + Radix) in `src/components/ui/`.

**Backend** (`server.js` + `api/`): Express server with JWT auth middleware (`authenticateToken`). Each API endpoint is a single file exporting `default async function handler(req, res)`. Routes are registered in `server.js` — new handlers must be added there to be reachable.

**Route registration** uses two patterns in server.js:
- Dynamic: `loadApiRoute('path/file.js')` inside an async route handler (most routes)
- Direct: `(await import('./api/path/file.js')).default` as the handler (campaigns block ~line 400)

Both work identically. Follow whichever pattern the surrounding routes use.

**Vite proxy**: `/api/*` requests proxy to `localhost:3003` with 180s timeout (configured in vite.config.js).

## Key Subsystems

**API Key Resolution** (`api/lib/getUserKeys.js`): Queries `user_api_keys` table, falls back to server env vars if user email matches `OWNER_EMAIL`. Route handlers should call `getUserKeys()` instead of reading `process.env` directly for AI provider keys.

**Model Registry** (`api/lib/modelRegistry.js`): Declarative config for all AI models — 8 image models (Nano Banana 2, Flux 2, SeedDream v4.5, Imagen 4, Kling Image v3, Grok Imagine, Ideogram v2, Wavespeed) and 14 video models (Kling 2.0 Master, Kling V3 Pro, Kling O3 Pro, Veo 2, Veo 3.1, Veo 3.1 Lite, Wan 2.5, Wan Pro, PixVerse v4.5, PixVerse V6, PixVerse C1, Hailuo/MiniMax, Grok Video, Wavespeed WAN). Each entry has `provider`, `endpoint`, `buildBody()`, `parseResult()`, and `pollConfig`. New models go here, not in if-else chains. PixVerse C1 (`pixverse-c1`) is R2V-capable — `isR2VCapable: true`, R2V endpoint at `fal-ai/pixverse/c1/reference-to-video`, accepts `image_references` array with typed refs (`subject`/`background`), uses `generate_audio_switch` (not `generate_audio`), cheapest video model at ~$0.005/sec.

**Media Generator** (`api/lib/mediaGenerator.js`): Generic dispatcher — `generateImageV2()` and `animateImageV2()` look up the model in the registry and handle provider-specific auth/polling. Replaces ~360 lines of per-model branching.

**Pipeline Helpers** (`api/lib/pipelineHelpers.js`): Shared polling logic for Wavespeed and FAL.ai async generation. `generateImage()` and `animateImage()` are backward-compat wrappers that delegate to `mediaGenerator.js`. Also exports `pollWavespeedRequest()`, `pollFalQueue()`, `uploadUrlToSupabase()`, `assembleShort()`, `assembleCarouselVideo()`, `assembleCarouselSlideshow()`, `generateMusic()`, `extractFirstFrame()`, `extractLastFrame()`. `assembleShort()` accepts a `musicVolume` parameter (default 0.15). `generateMusic()` defaults to ElevenLabs Music (`elevenlabs` via `fal-ai/elevenlabs/music`), always instrumental. No fallback — if ElevenLabs fails, returns null. `assembleCarouselSlideshow()` creates a video from static images (no AI video generation) via FFmpeg compose using `type: 'image'` tracks.

**Video Style Frameworks** (`api/lib/videoStyleFrameworks.js`): LEGACY — 76 structural frameworks. Superseded by the Production Engine's Fichtean Curve beat structure + niche blueprints for the Shorts Builder. Still exists for backward compat with old drafts. Do NOT use for new features.

**Shorts Builder** (`src/pages/ShortsBuilderPage.jsx` + `api/workbench/workbench.js`): Interactive 5-step Shorts creator at `/shorts/workbench`. Route `/shorts/builder` redirects there. 23 niches (20 original + DIY/Crafts, Parenting, Crypto).

Step 1 — Script: Niche selection → topic (3-level progressive funnel per niche in `TOPIC_SUGGESTIONS` inline data, or custom topic, or research via `/api/campaigns/research`). Generate Script calls the Production Engine (`api/lib/productionEngine.js`) which uses **Claude Sonnet 4.6** via Anthropic SDK (`@anthropic-ai/sdk`, tool_use for structured output) to produce a full **Fichtean Curve production package**: 6-7 beats (hook, context, escalation_1, escalation_2, climax, kicker, optional buffer), each with voiceover text, 1-4 visual prompts with camera directions, SFX cues, text overlays, transition types, music events. Also returns: global visual style, music direction (style, BPM, energy curve, key moments), title, hashtags, loop note, character references, total word count. The production engine injects the niche blueprint (`api/lib/nicheBlueprints.js`) into the system prompt — hook formula bank, visual style, visual guidance per beat, music/SFX palette, pacing, failure modes, niche-specific connector flavors. The system prompt contains extensive writing craft rules: sentence-level constraints (under 15 words, no semicolons, no "which" clauses), hook rules (must contain contradiction/paradox), context rules (deepen mystery, never explain hook), climax rules (3+ sentences with [BEAT] pauses), kicker rules (no CTAs, no fourth-wall breaks, quotable last line), and DELETE TEST enforcement. The old framework system (embedding matching, `builderFrameworks.js`, `frameworkMatcher.js`) is fully replaced — the Fichtean Curve IS the universal structure, niche blueprints provide the content guidance. Frontend displays beats with color-coded pills, visual prompt cards, SFX tags, music badges, transitions, loop notes. `ANTHROPIC_API_KEY` env var required.

Step 2 — Voice & Audio: Voice provider selection (Gemini TTS or ElevenLabs). 18 Gemini voices + 15 ElevenLabs voices shown. Each voice card has a ▶ preview button (`POST /api/voice/preview`). Voice style presets per niche. `[BEAT]` markers in narration are converted to `...` pauses for TTS. Generate Voiceover → TTS via `/api/workbench/voiceover`. Generate Music → production package's `music.style + music.energy_curve` passed as prompt to ElevenLabs Music via `/api/workbench/music`. Generate SFX → per-beat: each beat's `sfx_cues` combined into a prompt, individual 2-6s SFX clips generated via `/api/workbench/sfx` (accepts custom `prompt` param), stored with timestamp offsets for positioned placement during assembly. Timing alignment via Whisper word timestamps.

Step 3 — Visuals: Continuity mode (Continuous = last frame chains to next scene, Exciting = fresh image per scene but consistent characters). Video model selection (any I2V model). Image model for keyframes. Visual style from 123-style StyleGrid. Brand kit optional.

Step 4 — Captions: Caption style, position, highlight color, or disabled.

Step 5 — Generate: The production package's visual prompts are flattened — each visual prompt within each beat becomes one scene. A 7-beat script with 1-3 visual prompts per beat produces ~10-14 scenes. Per-scene pipeline:
1. **Cohesive Prompt Builder** (`api/prompt/build-cohesive.js`, tool='shorts') — receives the scene's `visualDescription` (PRIMARY creative direction from production package) + `previousSceneAnalysis` (ENHANCEMENT from previous clip's vision analysis) + visual style + camera direction + brand kit + niche mood + `continuityMode` + `characterReferences`. Mode-aware: Continuous mode = full visual continuity (same environment, lighting, characters); Exciting mode = fresh composition, only character appearance carried over. Character reference blocks in `[role: description]` format are preserved VERBATIM — the LLM enhances the scene around them but never paraphrases the character description.
2. **Generate starting image** — from cohesive prompt via image model. In Continuous mode for scene 2+, uses `lastFrameUrl` from previous clip instead.
3. **I2V (Image-to-Video)** — keyframe image + cohesive prompt sent to video model. Duration from `duration_hint_seconds` in visual prompt. `generate_audio: false` always.
4. **Save clip to Supabase** — immediately after generation (safety net, FAL CDN URLs expire in 24h). Via `/api/library/save`.
5. **Extract last frame** — `generate-clip` endpoint (`api/workbench/workbench.js`) automatically extracts last frame via `extractLastFrame()` and uploads to Supabase. Returns `last_frame_url`.
6. **Vision analysis** — `generate-clip` also runs GPT-4.1-mini vision analysis via `analyzeFrameContinuity()` on the last frame. Describes people (age, build, skin tone, hair, clothing, features), environment, lighting, camera. If `character_references` are provided, checks against them for consistency. Returns `vision_analysis`.
7. **Chain to next scene** — `last_frame_url` and `vision_analysis` feed into Step 1 of the next scene's Cohesive Prompt Builder call.
8. **Auto-save draft** — after each clip, full state (all config + all generated assets + clips so far) is saved to Supabase via `/api/workbench/save-draft`. If the window closes, progress is preserved up to the last completed clip.

Assembly: All clips + voiceover + music + positioned SFX tracks + music event volume automation → FFmpeg Compose API (`fal-ai/ffmpeg-api/compose`). Music events from production package beats (dropout = silence, riser = volume up, bass_drop = silence→hit, texture_change = volume dip) are converted to timestamped volume keyframes. Captions burned via `fal-ai/workflow-utilities/auto-subtitle` if enabled.

**Production Engine** (`api/lib/productionEngine.js`): Core script generator. Uses Claude Sonnet 4.6 (`claude-sonnet-4-6`) via `@anthropic-ai/sdk` with tool_use for Zod-validated structured output. Takes niche + topic + optional story context → returns production package. Zod schema defines: beats (voiceover, visual_prompts with camera/duration, text_overlay, sfx_cues, transition_out, music_event), music direction, visual_style, title, hashtags, character_references, loop_note, quality_checklist. `productionPackageToLegacyScript()` converts to old scene format for backward compat. `ANTHROPIC_API_KEY` env var required (not through `getUserKeys()` — it's a core infrastructure key).

**Niche Blueprints** (`api/lib/nicheBlueprints.js`): 23 niche production blueprints. Each has: `hook_formulas` (5 examples), `visual_style` (global prompt string), `visual_guidance` (per beat: hook, context, escalation, climax, kicker), `music` (style, bpm, energy curve), `sfx_palette`, `pacing`, `failure_modes`, `tone`, `archetype`. Also exports: `BEAT_STRUCTURE` (7 Fichtean Curve beats with purpose/rules), `ESCALATION_CONNECTORS`, `REFRAME_CONNECTORS`, `BANNED_PHRASES`, `VISUAL_TRANSITIONS` (8 types), `AUDIO_TRANSITIONS` (5 types), `SFX_BY_MOOD`, `NICHE_CONNECTOR_FLAVORS` (per-niche connector variants). `buildNichePromptSection(nicheKey)` assembles the niche-specific section of the production engine's system prompt.

**Shorts Templates** (`api/lib/shortsTemplates.js`): LEGACY niche definitions (23 niche types). Still used by the research endpoint and music mood defaults. The Production Engine uses `nicheBlueprints.js` instead for script generation.

**Legacy Script Generator** (`api/lib/scriptGenerator.js` + `api/lib/narrativeGenerator.js`): LEGACY — used by old workbench drafts only. The Production Engine (`productionEngine.js`) replaces this entirely. The preview-script endpoint (`api/campaigns/preview-script.js`) defaults to the production engine; pass `engine: 'legacy'` to use the old generator.

**Visual Styles** (`api/lib/visualStyles.js`): 14 visual style presets, Shorts-only. Frontend mirror: `src/lib/visualStylePresets.js`. The shared StyleGrid uses `src/lib/stylePresets.js` (123 styles) which is a separate, larger preset system — see Style Presets below.

**Topic Suggestions**: Inline in `ShortsBuilderPage.jsx` as `TOPIC_SUGGESTIONS` object. 3-level progressive funnel per niche (Category → Angle → Hook). All 23 niches have custom funnels. The old `src/lib/topicSuggestions.js` file may still exist but is not used by the Builder.

**Brand Mode** (toggle in Step 1 of Shorts Builder): Adapts the production engine for brand content marketing. When ON, replaces the niche grid with brand profile selection — the brand profile defines the niche, audience, and content strategy. The brand name NEVER appears in generated content.

Core concept: The brand's expertise is demonstrated through the quality of the insight, never stated through a pitch. Every video should be genuinely useful, shareable content that makes the viewer feel the exact problem the brand solves.

Components:
- **Brand Profiles** (`brand_profiles` table): Linked to `brand_kit` via `brand_kit_id`. Fields: `brand_domain`, `brand_expertise`, `target_viewer`, `target_viewer_pain_points[]`, `emotional_endpoint`, `primary_niche`, `secondary_niches[]`, `tone_override`, `voice_samples[]`, `hard_rules[]`, `content_angles` (JSONB array). Created automatically when saving a Brand Kit with Strategy tab fields filled. API: `POST/GET /api/brands/profiles`, `GET /api/brands/profiles/:id`.
- **Content Angles** (stored in `content_angles` JSONB): 3-5 per brand. Each has: `id`, `name`, `emotional_driver` (fear/identity/curiosity/injustice/wonder), `lens`, `description`, `endpoint` (per-angle emotional endpoint), `preferred_niche` (overrides brand's primary niche for this angle), `hook_patterns[]`, `weight` (percentage, all should sum to ~100). Weight-proportional rotation via `selectNextAngle()` prevents content angle fatigue.
- **Emotional Drivers** (`api/lib/brandMode.js`): 5 drivers with psychological grounding — Fear (Kahneman's prospect theory), Identity (Belk's self-concept), Curiosity (Loewenstein's information gap), Injustice (Haidt's moral foundations), Wonder (Keltner & Haidt's awe). Each defines: `tension_mechanic`, `script_instruction`, `pacing_note`, `failure_modes[]`.
- **Brand Writing Principles** (`api/lib/brandWritingPrinciples.js`): 8 principles injected into the system prompt in brand mode: (1) Depth is the brand voice, (2) The Gap — let the viewer close it, (3) Sentence rhythm matches emotional driver, (4) Use target viewer's language, (5) Emotional endpoint felt never stated, (6) Hooks must be shareable not just scroll-stopping, (7) Never burn the credibility, (8) Kicker is a choice not a conclusion. Includes a Brand Writing Quality Gate checklist that stacks with the main quality checklist.
- **Brand Context Builder** (`buildBrandContextSection()` in `brandMode.js`): Builds the brand-specific system prompt section injected after the niche blueprint. Includes: brand domain, expertise, target viewer, pain points, content angle details, emotional driver mechanics, hard rules, voice samples.
- **Post-generation Validation** (`validateBrandContent()` in `brandMode.js`): Scans `narration_full` for violations — brand name mentions, CTA patterns (40+ patterns: "link in bio", "sign up", "get started", etc.), selling language ("our product", "we offer", etc.). Returns array of `{ type, severity, message, match }`. Shown as red warning banner in the UI.
- **Brand Topic Generation** (`api/brands/generate-topics.js`): `POST /api/brands/generate-topics` — takes brand profile + content angle, generates 8-10 topics at the intersection of domain expertise × content angle × pain points × current news (via Exa API). Returns structured topics with title, hook concept, payoff, emotional endpoint, story context, evergreen flag. Uses GPT-4.1 + Zod structured output.
- **Auto-populate from URL**: Brand Kit URL extraction (`api/brand/extract-url.js`) now also extracts all Strategy tab fields — `brand_domain`, `brand_expertise`, `target_viewer`, `target_viewer_pain_points`, `emotional_endpoint`, `primary_niche`, `tone_override`, and 3-5 `suggested_content_angles` with emotional drivers and hook patterns. One URL → entire brand profile auto-filled.
- **Brand Kit Strategy Tab**: New "Strategy" tab (purple) in `BrandKitModal.jsx` with fields for all brand profile data + content angle builder (add/remove angles, set emotional drivers, weights, hook patterns). Saving a Brand Kit with strategy fields auto-creates/updates the linked `brand_profiles` record.

Brand Mode pipeline flow: Toggle ON → select brand profile (auto-sets niche) → select content angle or auto-rotate → generate topics → select topic → Generate Script (production engine receives brand context + angle + writing principles) → rest of pipeline identical.

DB tables: `brand_profiles`, `brand_topics`. Migrations applied via Supabase MCP.

**Audio Library** (`api/audio/library.js` + `audio_library` table): Reusable SFX and music tracks to avoid duplicate generation costs. Before generating new SFX/music, the system searches the library by niche + full-text keyword match. If a match is found, it's reused (free) and `use_count` is incremented. If no match, a new track is generated and auto-saved with extracted tags (60+ audio-specific keywords). Uses PostgreSQL `tsvector` full-text search via `search_audio_library()` RPC. API: `POST /api/audio/library` with actions: `search`, `save`, `increment`. Both SFX and music use the same table (`type` field distinguishes them).

**Draft System**: Save Draft persists ALL state including generated assets (script with full production package, voiceoverUrl, musicUrl, sfxUrl, SFX tracks with offsets, timingBlocks, generatedClips, finalVideoUrl, currentStep, all config). Load Draft restores everything and jumps to the saved step. Auto-save every 60s when meaningful state exists. Auto-save after each clip during generation pipeline. Draft list dropdown in header shows recent drafts with topic, niche, status, date.

**Workflow Engine** (`api/lib/workflowEngine.js`): Persistent state machine for long-running jobs (article→video pipeline). Steps: scrape → analyze → match_templates → create_campaign → generate_assets → concat → upload → finalize. State stored in `jobs` table. Supports pause/resume/retry.

**Voiceover Generator** (`api/lib/voiceoverGenerator.js`): Two TTS backends — Gemini TTS via `fal-ai/gemini-tts` (30 voices, `generateGeminiVoiceover()`) and legacy ElevenLabs via `fal-ai/elevenlabs/tts/eleven-v3` (`generateVoiceover()`). Gemini is the default for new Shorts. 30 Gemini voices exported as `GEMINI_VOICES`. Frontend mirror: `src/lib/geminiVoices.js` with `FEATURED_VOICES`.

**Caption System** (`api/lib/captionBurner.js`): `burnCaptions(videoUrl, captionConfig, falKey, supabase)` accepts either a full config object (font, size, color, stroke, position, animation, etc.) or a legacy string preset key (word_pop, karaoke_glow, word_highlight, news_ticker). Uses `fal-ai/workflow-utilities/auto-subtitle`. Presets exported as `CAPTION_STYLES`. API: `GET /api/styles/captions`.

**Scene Repair** (`api/shorts/repair-scene.js` + `api/shorts/reassemble.js`): Two endpoints for post-generation scene fixing. `repair-scene` uses Veo 3.1 First-Last-Frame (`fal-ai/veo3.1/fast/first-last-frame-to-video`) with adjacent scene frames; falls back to `animateImageV2` for last-scene edge case. `reassemble` rebuilds the final video from existing scene assets with re-captioning. Both look up jobs via `draft_id → campaign_id → jobs (input_json JSONB)`.

**Scheduled Publisher** (`api/lib/scheduledPublisher.js`): Polls for drafts with scheduled publish times and pushes to YouTube.

**Storyboards** (`api/storyboard/projects.js` + `api/storyboard/generate.js` + `StoryboardWorkspace.jsx` + `StoryboardList.jsx`): Persistent, database-backed video planning and production at `/storyboards` and `/storyboards/:id`. Settings-first flow: create storyboard (name only, no frames) → configure settings (duration, style, mood, model, brand, characters) → Generate Script (AI decides scene count via `calculateSceneCount()`, creates frames from GPT output with variable durations) → generate preview images → inline edit/lock/split/delete scenes → export PDF → share public review link (`/api/storyboard/review/:token`, no auth) → approve for production. Three tabs: Settings (default when no scenes), Storyboard (frame grid + detail panel), Production (video generation status). Frames are NOT pre-created — they only exist after script generation. Script generation: Stage 1 narrative (`storyboardNarrativeGenerator.js`) → Stage 2 visual director (`storyboardVisualDirector.js`), both via GPT-4.1 mini + Zod structured output. Scene management: split scene (halves duration, new scene gets empty content), delete scene (renumbers + recalculates timestamps), lock scene (prevents regeneration). DB tables: `storyboards` + `storyboard_frames`. Migration: `migration-storyboard-system.sql`. CRUD API uses path-based routing in `projects.js` — the `app.all('/api/storyboard/projects*')` catch-all handles all routes. Script + preview generation routes registered BEFORE the catch-all in server.js. The old `StoryboardPlannerWizard.jsx` and `StoryboardEditor.jsx` are dead code — `StoryboardWorkspace.jsx` is the active page.

**JumpStart** (`api/jumpstart/`): Image/video generation endpoints used by the Video Ad Creator and Storyboards. 6 endpoints: `generate` (FormData with image blob), `result` (poll async), `save-video`, `edit`, `extend`, `erase`. Models are selected client-side and passed as form field — the backend dispatches to the appropriate provider. Supports Veo 3.1 variants (reference-to-video, fast, first-last-frame), Veo 3.1 Lite variants (I2V, first-last-frame — 60% cheaper), PixVerse V6 (audio via `generate_audio_switch`), Kling O3 R2V (character references), Kling O3 V2V (video-to-video restyle/refinement), Seedance, Grok (image-to-video, edit, extend), and Wavespeed WAN. The `extend` endpoint supports 3 model families: Seedance 1.5 Pro (Wavespeed, 4-12s), Veo 3.1 Fast Extend (FAL, fixed 7s), and Grok Imagine Extend (FAL, 2-10s). Frontend: `JumpStartVideoStudioModal.jsx` provides the extend/edit UI.

**Imagineer** (`src/components/modals/ImagineerModal.jsx` + `api/imagineer/`): AI image generation and editing suite with two modes — Text-to-Image (4-step wizard: Subject → Style → Enhance → Output) and Image-to-Image (edit existing images). Both modes support multi-style bulk create via `StyleGrid multiple` — selecting multiple styles generates one image per style with a multi-result grid (save/retry/use per result). T2I `artisticStyle` and I2I `i2iStyle` are both arrays. T2I models: Nano Banana 2, Seedream v4, Flux 2 (LoRA). I2I models add Wavespeed Nano Ultra and Qwen Image Edit for multi-image blending. Has its own 100+ style presets map in `generate.js` (separate from `api/lib/visualStyles.js` which is Shorts-only). Reference images are analyzed via GPT-4.1 mini vision (`describe-character.js`). All structured inputs are assembled into prompts by the shared Cohesive Prompt Builder.

**Imagineer I2I Multi-Image**: Nano Banana 2, Seedream, Wavespeed Nano Ultra, and Qwen all support multiple `image_urls` in I2I mode (e.g., scene backdrop + character → composite). `api/imagineer/edit.js` accepts both `image_url` (single, backward-compat) and `image_urls` (array). Wavespeed models route to `/api/images/edit`; FAL models (nano-banana-2, seedream, fal-flux) route to `/api/imagineer/edit`.

**Turnaround Sheet** (`api/imagineer/turnaround.js` + `TurnaroundSheetWizard.jsx`): Dynamic-grid character turnaround sheets. 8 pose set presets in `api/lib/turnaroundPoseSets.js` (frontend mirror: `src/lib/turnaroundPoseSets.js`). Grid dimensions are derived from the pose set structure (`rows.length` × `rows[0].cells.length`): legacy sets are 4×6 (24 cells), 3D-optimized sets are 2×2 (4 cells, 6× more pixels per cell), R2V-optimized set is 3×2 (6 cells). Three grid aspect ratios: 2×2 → 1:1 square, 3×2 → 3:2 landscape (2560×1440 Seedream, 1536×1024 Flux), 4×6 → 2:3 portrait. Pose sets: "3D Reference — Angles" (orthographic), "3D Reference — Action" (dynamic), "R2V Reference" (3×2 grid: full body front/3-4/side + matching portrait close-ups per angle — optimized for Reference-to-Video workflows), "Standard 24" (classic turnaround, Row 6 has per-angle portrait close-ups: Face Front, Face 3/4, Face Side, Hand Detail), "Expressions Focus", "Action Heavy", "Fashion/Outfit", "Creature/Non-Human". `getPoseSetGrid(id)` returns `{ cols, rows, total }`. 6-step wizard: Character → Style & Model → Props → Refinements → Results → Cell Editor. Generation: `characters × styles × poseSets` cartesian product, 4-concurrent limit. Edit models use synchronous `fal.run` with fallback chain; generate models use `queue.fal.run`. Edit model results are auto-upscaled 2× via Topaz (`fal-ai/topaz/upscale/image`, PNG, Standard V2) before returning to the frontend. Three background modes: "White Background" (default), "Gray Background" (industry standard for production sheets, avoids Veo 3.1 R2V issues with pure white), "Scene Environment" (contextual backgrounds for R2V-compatible references). Controlled via `backgroundMode`/`sceneEnvironment` in `buildTurnaroundPrompt()`. Prompt uses identity-first hierarchy: avoidNote → character identity → reference instructions → layout/background → consistency/alignment instructions → style → props → row definitions → taxonomy tags → avoidNote repeat.

## Gotchas

- `fal-ai/nano-banana-2` (T2I generation) and `fal-ai/nano-banana-2/edit` (I2I composition with `image_urls[]`) are separate FAL endpoints. Same applies to Seedream. Don't confuse them — `api/imagineer/generate.js` hits the base endpoint, `api/imagineer/edit.js` hits the `/edit` variant.
- Tag tables (`image_tags`, `image_tag_links`) require migration `supabase-migration-tags.sql`. Without it, the library shows 404/500 errors on tag queries.
- `.env` lines must each have their own newline — concatenated lines silently break dotenv parsing.
- `api/lora/seed-library.js` is a CLI script (`node api/lora/seed-library.js`), not an Express handler. Don't register it as a route.
- Migrations are loose SQL files at project root (`supabase-migration.sql`, `supabase-migration-v2.sql` through `v8`, plus `-tags`, `-storyboard-presets`, `-shorts-overhaul`, `-linkedin`, `-app-source`, `-library-metadata`, `-proposal-media`, `-storyboard-system`, `-carousel`, `-carousel-video`, `-ads`), in `api/migrations/`, and in `supabase/migrations/` (newer, date-prefixed format like `20260401_lora_training_model.sql`). Not managed by Supabase CLI.
- Vite dev server runs on port 4390 (not the default 5173).
- LoRA training models use different field names for the zip URL: `images_data_url` (Flux Fast, Hunyuan), `image_data_url` (Flux Kontext, Qwen, Z-Image), `training_data_url` (Wan models). The `trainingModelRegistry.js` handles this via `buildBody()` — don't construct FAL payloads manually.
- LoRA training models also differ on trigger word field: `trigger_word` (Flux Fast, Hunyuan) vs `trigger_phrase` (Flux Portrait, Wan, Qwen). Again handled by the registry.
- `brand_loras.training_model` stores the registry ID (e.g. `'flux-lora-fast'`, `'wan-22-image'`). Used by `result.js` to resolve the correct FAL endpoint for polling and the correct `parseResult()` for extracting the model URL.
- Some routes return immediately and do background work (e.g., `generate-thumbnails`). Check for `res.json()` before async blocks.
- Webhook routes (`/api/webhooks/content`, `/api/article/from-url`, `/api/article/bulk`) skip auth — they use webhook secrets or brand_username verification instead. OAuth callback routes (`/api/linkedin/oauth/callback`, `/api/meta/callback`, `/api/tiktok/callback`, `/api/youtube/callback`, `/api/ads/google/callback`) also skip auth — they're redirect targets from external OAuth providers.
- `platform_connections` is the unified token store for all platforms. Legacy tables (`brand_youtube_tokens`, `linkedin_config`) are still read as fallback during migration. When adding new publishing features, always resolve tokens via `tokenManager.loadTokens()` first.
- Meta OAuth stores nonces with platform `'meta'` (not `'instagram'` or `'facebook'`), since a single OAuth flow discovers both platforms.
- All Veo 3.1 endpoints (`generate.js`, `extend.js`) use `auto_fix: true` so FAL rewrites prompts that trigger content policy (copyrighted brand names). The Cohesive Prompt Builder's storyboard system prompt also strips brand names (Pixar, Disney, DreamWorks, Cocomelon, etc.) proactively. All style preset files are now brand-free — brand names have been stripped from `videoStylePresets.js`, `stylePresets.js`, `visualStyles.js`, `visualStylePresets.js`, and `api/imagineer/generate.js`. If Veo R2V returns 422 "no_media_generated", check for brand names in the prompt.
- FAL CDN URLs (`v3b.fal.media`, `fal.media`) are temporary and expire within 24 hours. Always upload generated media to Supabase storage via `uploadUrlToSupabase()` or the `/api/library/save` endpoint before storing URLs in the database or displaying them in production pages. The Shorts Builder auto-saves each clip to Supabase immediately after generation (via `/api/library/save` in the frontend pipeline).
- Video model duration formats differ by provider: Veo 3.1 and Veo 3.1 Lite accept ONLY `'4s'`, `'6s'`, `'8s'` — any other value (including `'5s'`, `'7s'`) causes a 422 error. The `veoDuration()` function in modelRegistry.js maps: ≤4→4s, ≤6→6s, else→8s. Kling/Wan/PixVerse use `"5"`/`"10"` (string number), Wavespeed uses integer `5`/`8`, some models (Hailuo, Wan Pro) don't accept duration at all. The model registry handles this — don't hardcode duration format.
- `/shorts/new` and `/shorts/builder` both redirect to `/shorts/workbench` (the Shorts Builder, `ShortsBuilderPage.jsx`). The old Shorts wizard in `CampaignsNewPage.jsx` has been removed — that page now only handles Ad campaigns. `ShortsWizardPage.jsx` and the old `ShortsWorkbenchPage.jsx` are dead code. Draft review at `/shorts/draft/:draftId`. Workbench drafts saved via `/api/workbench/save-draft`.
- `/templates` route exists (`TemplatesPage.jsx`) — accessible from the sidebar Social Tools section. `JumpStartVideoStudioModal.jsx` and `MotionTransferModal.jsx` are additional modals in `src/components/modals/` used by the Video Ad Creator.

- `api/lib/modelRegistry.js` image models use either `image_size` (Flux, SeeDream, Ideogram) or `aspect_ratio` (Imagen4, Kling Image, Grok) — check the registry's `buildBody()` before adding new models.

The actual provider check is `isWavespeedModel = i2iModel.startsWith('wavespeed-')` on line ~465. When debugging I2I routing, check both variables.
